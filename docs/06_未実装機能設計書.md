## 1. Firestore ルールのデプロイ（メンバー管理の有効化）【運用作業】

### 背景
設定パネルの「メンバー管理」（`data/allowedEmails` による許可リスト）は実装済みだが、リポジトリの `firestore.rules` を Firebase コンソールへ手動デプロイするまで**サーバ側では有効にならない**。デプロイ前でもオーナー（kei.n412@gmail.com）の利用と既存機能はすべて従来どおり動く。

### 作業手順
1. Firebase コンソール → Firestore Database → ルール を開く
2. リポジトリ直下の `firestore.rules` の内容を貼り付けて「公開」
3. アプリの設定パネル → メンバー管理 でメンバーの Gmail を追加して動作確認（追加メンバーで一度サインインし、データが見えること）

### 注意
- ルール内のオーナーメールと `src/firebase.js` の `OWNER_EMAILS` は**必ず同じ値を保つ**（ロックアウト防止のための二重化）
- 許可リストドキュメント（`allowedEmails`）の書き換えはオーナーのみ可能（ルールで制限済み）

## 2. 請求書ステータスと売上登録表の自動連携

### 背景
現在は「請求書のステータス（送付済み・入金済み＋日付）」と「売上行の 12.請求書送付日／13.入金確認日」を**それぞれ手入力**しており、二重入力になっている。売上側の「請求・入金チェック」カード（実装済み）は売上行の日付だけを見るため、請求書側で送付済みにしても売上側は未送付扱いのまま。

### 仕様案
1. **紐付けデータ**：請求書ドキュメントに `salesRowIds: string[]`（紐付く売上行の id）を追加する
2. **紐付けUI**：請求書エディタの基本情報タブに「売上行と紐付け」セクションを追加
   - 宛先会社名と発行月（±1か月）で売上行を候補表示し、チェックボックスで複数選択
   - 「案件から自動入力」した場合は同じ案件名の行を自動候補にする
3. **同期方向は 請求書 → 売上 の一方向のみ**（逆方向は行わない。手動行の値を勝手に書き換えない原則を守る）
   - ステータスを「送付済み」にした時：紐付いた売上行の `invoiceSentDate` が**空なら**請求書の送付日を書き込む
   - 「入金済み」にした時：`paymentConfirmedDate` が**空なら**入金日を書き込む
   - 既に値がある行は上書きしない（confirm で上書き可にしてもよい）
4. **請求金額の突合**：紐付けた売上行の税込合計と請求書合計が一致しない場合、エディタ内に注意表示（保存は妨げない）

### 影響範囲
- `src/billing/BillingView.jsx`（エディタUI・ステータス変更処理）
- `src/billing/billingUtils.js`（`blankDoc` に `salesRowIds: []`）
- 売上行の更新は `salesStore.set(ym, monthData)` で該当月のみ書き込む
- 売上行の id は月をまたいで移動しない前提（自動同期生成行は `srcRound` 固定）

### 受け入れ条件
- 請求書を送付済みにすると、紐付けた売上行の請求書送付日が入り、売上側「請求・入金チェック」の未送付件数が減る
- 紐付けなしの請求書は従来どおり動く（後方互換）

## 3. 見積書「工程予定表」の作業項目ラベル編集

### 背景
工程予定表の作業項目は `SCHEDULE_PROCESS_COLUMNS` の固定7列（アングル確認パース提出・初稿パース提出 など）。実際の運用（Google Drive 上の見積書）では案件によって「ホワイトパース提出」「色付きパース提出」「チェックバック・仕上げ関連の資料送付」等にラベルを書き換えて使っており、アプリでは再現できない。

### 仕様案
1. スケジュール列データ `{ label, party, date, times }` の `label` と `party`（client/us）をエディタで編集可能にする
   - 既にデータ構造は列ごとに `label`/`party` を保持しているため、**移行は不要**
2. エディタ（`ScheduleEditor`）の各列ヘッダーに、ラベルのテキスト入力と「お客様/当社」切り替えを追加
3. 列の追加・削除ボタン（最大10列程度・最小1列）
4. 「既定の7工程に戻す」ボタン（`SCHEDULE_PROCESS_COLUMNS` から再生成。日付・時間はクリアされる旨を confirm）

### 影響範囲
- `src/billing/BillingView.jsx` の `ScheduleEditor` のみ（印字側 `SchedulePage` は既に `c.label`/`c.party` を参照しているため変更不要）

## 4. 見積書「アングル」ページへの参考画像貼り付け

### 背景
見積書4枚目のアングル欄は現在テキストメモのみ（UI にも「今後対応予定」と表示中）。実運用ではアングル指示のスクリーンショットを貼っている。

### 仕様案
1. **保存先**：Firebase Storage は現在未使用のため、まずは**画像を縮小して dataURL 化し Firestore に保存**する方式とする
   - 貼り付け時に canvas で長辺 1200px・JPEG 品質 0.8 に縮小（1枚あたり概ね 100〜300KB）
   - 1帳票 = 1ドキュメント化済みのため、外観・内観 各1枚（計2枚）までなら 1MB 制限内に収まる
   - 将来枚数を増やす場合は Firebase Storage（`workspaces/{wid}/billing/{docId}/…`）へ移行し、ルール整備を行う
2. **UI**：アングルタブに画像のドラッグ&ドロップ／ペースト（clipboard）／ファイル選択。プレビューと削除ボタン
3. **データ**：`doc.angles.exteriorImage` / `doc.angles.interiorImage`（dataURL文字列 or ''）
4. **印字**：`AnglePage` の枠内に画像を `max-width:100%; max-height:100%; object-fit:contain` で描画。画像とテキストの併記可

### 制約・注意
- dataURL 方式は Firestore ドキュメント 1MB 制限が上限。貼り付け時に合計サイズを検査し、超える場合はエラー表示
- 帳票の複製時は画像もコピーされる（サイズに注意）

## 6. 帳票の印字オプション（帯色・請求書の日付列）

### 背景
帳票フォーマット統一（2026-07-03）で、①上下の帯を全種別ダークグレーに統一、②請求書の明細から日付列を廃止した。運用してみて「種別ごとに帯色を分けたい」「請求書に日付列を残したい」となる可能性があるため、オプションとして設計しておく。

### 仕様案
1. storage キー `billingPrintOptions`（JSON: `{ coloredBands: boolean, invoiceDateColumn: boolean }`、既定はともに false）
2. 帳票一覧の「自社情報・振込先」パネルに印字オプションのチェックボックスを追加
3. `BillingDocument` にオプションを props で渡す
   - `coloredBands: true` → 帯と明細下線を `DOC_TYPES[].accent`（見積=黒・発注=青・請求=緑）に
   - `invoiceDateColumn: true` → 請求書の明細に日付列を復活（データは保持済みのため表示制御のみ）

### 備考
明細の日付データ（`item.date`）と編集UIは削除していないため、表示制御のみで実装できる。

## 7. 旧データのバックアップキー削除導線

### 背景
1ドキュメント集中保存からの移行（2026-07-03）で、旧データを `billingDocuments_backup` / `salesLedger_backup` キーに退避した。新方式で問題なく運用できた後は不要になるが、現在は Firebase コンソールから手で消すしかない。

### 仕様案
1. 設定パネルに「メンテナンス」セクションを追加（オーナーのみ表示）
2. バックアップキーの有無・サイズ・退避日時を表示し、「削除」ボタン（confirm 2段階：キー名の確認 → 最終確認）
3. 移行から一定期間（例：30日）経過するまでボタンを無効化して誤削除を防ぐ

### 対象キー
`billingDocuments_backup`、`salesLedger_backup`、（旧 `tasks` は移行時に削除済みのため対象外）

## 9. タスクメモのプッシュ通知

### 背景
タスクメモの開始時刻通知は「アプリを開いている間」のみ（`Notification` API＋アプリ内バナー）。タブを閉じている間は通知できない（コードコメントにも明記済み）。

### 仕様案（実装する場合）
1. Firebase Cloud Messaging（FCM）＋ Service Worker によるWebプッシュ
2. 配信トリガーに Cloud Functions（スケジュール実行で1分ごとに対象メモを検索）が必要
   - 現在の構成（静的ホスティング＋Firestore のみ・サーバレス関数なし）から**構成が大きく変わる**ため、優先度は低
3. 代替案：メモをGoogleカレンダーへ連携（`sync/Code.gs` の GAS を拡張）し、通知はカレンダーに任せる方式なら現行構成のまま実現可能

## 10. 担当者ごとの稼働時間設定（時差対応）

### 背景
稼働時間（午前・午後のスロット）は settings で**全員共通**。実際には日本側とベトナム側（JST−2時間）で稼働帯が異なり、タイムライン・終了予定・空き時間サマリーが実態とずれる。スケジュール精度（予測遅延の事前検知・おすすめ担当者提案の精度）に直結する。

### 仕様案
1. 従業員マスタに任意項目「稼働時間」を追加：`{ morningStart, morningEnd, afternoonStart, afternoonEnd }`（未設定なら全体設定を使用）
2. スケジューラの `getDaySlots` / `dayWorkSlots` を担当者引数付きに拡張し、担当者別スロットを参照
3. カレンダーの時間軸表示は「全担当者のスロットの和」を描画範囲にする
4. 簡易運用として「時差シフト（分）」1項目だけ持たせる案もある（全スロットを一律にずらす）

### 影響範囲
- `scheduleTasks` 系（`getDaySlots`・`dayFreeIntervals`・残業/不在との重なり処理）
- カレンダー各表示モード・空き時間サマリー・メッセージビューの時刻表記

### 注意
スケジューラの中核に手が入るため、`/tmp` の node スクリプトで前後の配置結果を比較検証してから反映する。

## 11. sync/Code.gs（シート同期）の旧キー書き込み問題【要確認】

### 背景
`sync/Code.gs`（スプレッドシート→Firestore同期）は旧形式の `workspaces/{wid}/data/tasks` キーに書き込む。アプリは 2026 年時点でタスクを 1件1ドキュメントのサブコレクション（`workspaces/{wid}/tasks/{taskId}`）で管理しており、起動時に旧キーを見つけると「サブコレクションが空の場合のみ取り込み → **旧キーを削除**」する。つまりサブコレクションに既にタスクがある通常運用では、**シート同期の結果は取り込まれずに削除される**。

### 対応案
- **同期をまだ使っている場合**：Code.gs を Firestore REST の `tasks` サブコレクションへの upsert（externalId ベースの重複防止は維持）に改修する。書き込み先：`projects/{pid}/databases/default/documents/workspaces/{wid}/tasks/{taskId}`
- **使っていない場合**：Code.gs 冒頭に「退役済み（アプリ直接登録に移行）」と明記するか、ファイルを削除する

### まず確認すること
スプレッドシート側のトリガー（時間主導型）が現在も有効かどうか。有効なら無駄なFirestore書き込みが定期的に発生している。
