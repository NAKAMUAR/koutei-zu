## 1. はじめに

### 1.1 本書の目的

本書は「工程図（koutei-zu）」が満たすべき業務上・機能上・非機能上の要件を定義することを目的とする。建築CG（パース・写真合成等）制作チームの日々の制作スケジュールを、案件・視点（依頼項目）・工程ステップの単位で管理し、担当者ごとの稼働を自動でタイムライン化することを狙いとする。

### 1.2 用語定義

| 用語 | 定義 |
|---|---|
| 案件（プロジェクト） | お客様から受注した制作の単位。社外案件名・社内案件名（案件コード）を持つ。 |
| 視点（依頼項目） | 案件内の制作対象の単位。例：「外観昼景」「写真合成」。担当者は視点ごとに割り当てる。 |
| ステップ（工程） | 視点内の作業工程。例：「ホワイト」「カラー」「その他修正」。各ステップが制作時間を持つ。 |
| タスク | 「1案件 × 1視点 × 1ステップ」で確定する管理上の最小単位。1タスク=1Firestoreドキュメント。 |
| 制作時間（hours） | そのステップの見積もり作業時間（h）。スケジュール枠の長さの基準。 |
| 完了時間（completedHours） | そのステップで既に完了した作業時間（h）。 |
| 開始時間指定（manualStart） | 自動スケジュールより優先してタスクの着手時刻を固定する指定。 |
| 終了時間指定（manualEnd） | 作業終了予定時刻の固定指定。エンジニアの作業完了見込み時刻。 |
| 納期（deadline） | お客様へ提出する期限日。視点ごとに設定する。 |
| 残業 | 担当者・期間・時間帯を指定して通常稼働枠に追加する稼働時間。 |
| 欠勤（休日・不在） | 担当者の対応不可日・時間帯。終日または時間帯で指定する。 |
| 差し込み | 開始時間を指定したタスクが時間枠を先取りし、指定なしのタスクが前後の空きへ分割して入ること。 |

### 1.3 対象範囲

- 対象：制作チームの社内スケジュール管理（案件登録・進捗・カレンダー可視化・連絡文生成）
- 対象外：受注管理・請求・会計、外部顧客向けの公開機能、LINE等の外部メッセージ自動連携（将来検討）

## 2. 業務要件

### 2.1 現状と課題

- 建築CG制作は「案件 → 視点 → 工程」の階層を持ち、担当者・優先順位・納期が絡むため、手作業のスケジュール調整が煩雑。
- 担当者ごとの稼働がいつ空くか、案件がいつ終わるかを即座に把握したい。
- 進捗（完了時間）に応じて終了見込みが変動するため、常に最新の予定を全員で共有したい。

### 2.2 業務フロー（あるべき姿）

1. 受注した案件を、視点・ステップ・制作時間とともに登録する（仮案件としての先行登録も可）。
2. 担当者・優先順位・開始/終了/納期を設定すると、システムが自動でタイムラインを生成する。
3. 日々の進捗（完了時間）を入力すると、終了見込みが自動で前後に調整される。
4. カレンダー・担当者別・サマリーで状況を共有し、会社別の業務連絡文を生成して送付する。
5. 完了した視点・案件は終了時刻（実績）を記録して完了タブへ移動する。

### 2.3 利用者（アクター）

| アクター | 説明 |
|---|---|
| 制作管理者 | 案件登録・優先順位・スケジュール調整・連絡文送付を行う中心利用者。 |
| 制作担当者（エンジニア） | 自分の担当タスクの進捗入力・確認を行う。 |

※ 認証は許可されたGoogleアカウント（Gmail）のみ。同一ワークスペースIDの利用者でデータを共有する小規模チーム運用を前提とする。

### 3.2 主要機能の詳細要件

#### F-02 新規タスク登録
- 案件は「社外案件名」を必須とする。社内案件名・会社名・お客様担当者は任意。
- 1案件に複数の視点を登録でき、各視点は複数のステップ（名称＋制作時間＋完了時間）を持つ。
- 視点ごとに担当者を上書き設定できる（未設定時はデフォルト担当者を適用）。
- 制作時間0でも登録可（先行枠取り）。完了時間≧制作時間のステップは登録時点で完了扱い。
- 「仮案件」フラグでスケジュール枠は確保しつつ仮予定として区別表示できる。

#### F-05 / F-06 自動スケジューリングと指定
- 営業時間は午前（既定8:00–12:00）＋午後（既定13:00–17:00）、土日除外、第2・第4土曜は午前のみ。
- 作業順は「案件の並び順 → 案件内の優先順位 → ホワイト工程を全視点優先 → 登録順」。
- 開始時間を指定したタスクは事前に枠を予約（差し込み）。指定なしタスクは前後の空きへ分割。
- 終了時間指定は「その時刻で打ち切り、後続タスクはその時刻以降に開始」。
- 優先順位は会社ごとに1から採番する。

#### F-07 納期超過警告
- 納期は視点ごとに日付で設定。登録/更新時に実スケジュールを試算し、視点の終了予定日が納期日を超える場合は確認モーダルと入力フォームのプレビューに警告を表示する。
- 案件ヘッダー・視点カードに納期バッジを表示し、超過時は赤系で警告表示する。
- 案件の納期表示は「視点ごとの納期のうち最も早いもの」を用いる。

#### F-08 進捗連動の終了予定調整（重要要件）
- 開始が現在より過去のタスクは「開始〜現在の経過稼働時間 ＋ 残作業時間（制作時間−完了時間）」で枠を取り直す。
- 例：4hの案件で現在10:00（8:00着手）、完了2h → 終了予定12:00のまま。完了1h → 終了予定14:00へ後ろ倒し。完了3h → 11:00へ前倒し。
- 1分ごとに再計算し、後続タスクの開始も追従する。

#### F-09 残業／F-10 欠勤
- 残業：担当者・期間（開始日〜終了日）・時間帯を指定し、通常稼働枠にマージしてスケジュール・カレンダーへ反映。
- 欠勤：終日または時間帯で対応不可を指定。該当時間はスケジュールから除外し、カレンダーにグレー表示。

#### F-14 終了超過ポップアップ
- 視点単位で終了予定を過ぎると表示。対応は以下4種。
  - ① 視点完了（実終了時刻を入力）
  - ② 追加修正（その視点に追加ステップを足す）
  - ③ 遅延（新しい終了予定へ更新し、差分の稼働時間を加算・遅延履歴を記録）
  - ④ 後で（30分スヌーズ）

## 4. 非機能要件

| 分類 | 要件 |
|---|---|
| 性能 | 数百件規模のタスクで、編集・スケジュール再計算が体感即時であること。 |
| 可用性 | GitHub Pages＋Firebaseのマネージドサービスに依存。SLAはサービス標準に準拠。 |
| 同時編集 | 1タスク=1ドキュメントの差分書き込みで、複数端末同時編集でも表示が乖離しないこと。 |
| リアルタイム性 | Firestoreの購読で、他端末の変更が即時に画面反映されること。 |
| セキュリティ | 認証必須。Firestoreルールで許可メール（email_verified）のみ読み書き可。 |
| 対応端末 | PC／タブレット／スマートフォン（レスポンシブ。ドラッグ操作はPC、↑↓ボタンはスマホ）。 |
| 国際化 | 日本語UI・日本のカレンダー慣習（土日・第2/4土曜）に準拠。 |
| 保守性 | React単一ファイル中心（`src/App.jsx`）。純粋ロジックはNodeで単体検証可能。 |

## 5. 制約・前提条件

- フロントエンドのみのSPA。専用サーバを持たず、バックエンドはFirebase（Firestore＋Auth）。
- TypeScript不使用（JavaScript/JSX）。スタイルはインラインスタイル。
- ホスティングはGitHub Pages。`npm run deploy`（vite build → gh-pages）で公開。
- ワークスペースIDで共有範囲を区切る単一テナント想定（マルチテナントの厳密分離は対象外）。
- 祝日カレンダーは未対応（土日・第2/4土曜のみ自動判定）。
